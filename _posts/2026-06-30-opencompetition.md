# 面向 RISC-V Vector 扩展的大语言模型推理并发调度优化

## 赛题完整描述：

<details>
<summary>展开查看赛题完整描述原文</summary>
<pre style="white-space: pre-wrap;">

RISC-V Vector 扩展（RVV 1.0）已被纳入 RVA23 Profile 强制规范（2024 年 10 月批准），正在成为 RISC-V 处理器的标准配置。2025 年初，llama.cpp 等主流大语言模型（LLM）推理框架完成了 RVV 1.0 适配，使得在 RISC-V 设备上运行 LLM 成为现实。

然而，RVV 的引入带来了值得单独研究的操作系统调度挑战。x86 AVX-512 与 ARM SVE 等架构同样存在扩展状态上下文管理问题，但 RVV 的变长寄存器（VLEN 由实现定义，最大可达数千 bit）与 Vector Length Agnostic（VLA）设计使其系统代价与管理复杂度显著不同。在RISC-V 架构中，每个 hart 拥有 32 个向量寄存器（v0–v31），单次完整 V 状态保存需要512 字节至 1024 字节以上的内存写入。Linux 通过 VS 状态位配合惰性上下文管理策略（lazy save / restore）降低不必要的状态切换开销，但在多向量密集型线程并发运行时，当向量状态为 dirty 的线程被抢占，仍会触发 V 寄存器组的完整保存，在 LLM 推理这类高强度持续向量计算场景下积累出不可忽视的调度开销。

LLM 推理是典型的高强度持续向量计算场景：prefill 阶段执行大规模矩阵乘（GEMM），decode阶段执行矩阵向量乘（GEMV），两个阶段均需长时间持续占用向量寄存器。在推理服务器同时处理多个并发请求时，Linux 默认调度策略（CFS）对向量密集型推理线程与普通线程一视同仁，导致推理线程在计算关键路径上被频繁抢占，积累的 V 状态切换开销造成整体吞吐下降与尾延迟（P99 TTFT）恶化。

目前，学术界和工业界均缺乏专门针对 RISC-V V 状态感知的 LLM 推理调度策略，这是一个兼具理论价值与工程意义的开放问题。本赛题以 RISC-V Vector 扩展为实验平台，其核心关注点——具有高代价扩展状态的处理器在 AI 推理负载下的 OS 调度协同——同样适用于 x86 AMX/AVX-512、ARM SVE2 等架构，参赛成果具备向其他平台迁移的理论基础。

## 🎯 核心问题与背景

- **技术演进**：RVV 1.0 已成为 RISC-V 处理器的标准配置，主流推理框架（如 `llama.cpp`）已适配 RVV 1.0。
- **性能瓶颈**：RVV 采用变长寄存器（VLA）设计，上下文体积庞大（512B - 1024B+）。在 LLM 推理的高强度向量计算（Prefill/Decode）场景下，Linux 默认的 CFS 调度器会频繁抢占持有 `dirty` 向量状态的推理线程，导致**大量的 V 状态保存/恢复开销**。
- **后果**：系统整体吞吐量（TPS）下降，尾延迟（P99 TTFT）恶化。

## 📋 赛题任务要求

### 1. 问题量化（必做）

- **手段**：利用内核 tracepoint、kprobe、eBPF 或 perf/PMU 等系统观测工具。
- **目标**：对 `riscv_vstate_save` 等关键路径插桩，量化并发推理下的 V 状态切换开销，建立其与并发请求数、模型规模的量化关系，并分析其对延迟和吞吐的影响。

### 2. 调度优化实现（必做，至少实现一种）

- **方案 A（入门级）**：结合 CPU affinity 与 cgroup cpuset，将推理线程池绑定至指定核心集合（如利用 `llama.cpp` 的 `--cpu-mask` 参数），减少跨核迁移与抢占。
- **方案 B（进阶级）**：基于 **sched_ext**（Linux 6.12+ 的 BPF 可扩展调度框架）实现 **V 状态感知调度器**；或修改现有接口优化持有 `dirty` V 状态线程的调度与放置，减少时间窗口内的切换次数。

### 3. 性能评测（必做）

- **基准软件**：`llama.cpp` server 模式。
- **模型与配置**：$\le$ 3B 参数量的量化模型（如 Llama-3.2-1B/3B Q4_K_M）；并发请求设置为 2、4、8；CPU 固定为 `performance` 模式。
- **硬件要求**：正式性能数据**必须来自真实 RVV 1.0 硬件**（如进江山 K1 / 🚀香蕉派 BPI-F3 / 牛奶创客 Jupiter，VLEN=256），每组至少重复 3 次并报告均值与标准差。*（QEMU 数据仅算功能验证，不计性能分）*
- **指标**：对比优化前后的推理吞吐（TPS）、平均 TTFT、P95 TTFT。

### 4. 可选完成项（加分项）

- 设计用户态-内核态的调度提示接口（如扩展 `sched_setattr` 或自定义 `ioctl`）。
- 与 x86（AVX-512）平台的向量切换机制进行横向对比分析。
- 向 `llama.cpp` 或 Linux 内核上游社区提交 Patch 或 RFC。
